# ITW Editor (iTaiwan Editor) 開發計畫

> **文件版本**: v2.1 | **最後更新**: 2026-04-16
> **專案代號**: iTaiwan | **核心格式**: `.itw`

## 專案願景

打造一種具備高度互操作性、人類可讀、且對 AI 極度友好的通用數據容器格式。透過系統底層的排版演算法與瀏覽器端的解析渲染，**將非結構化的「文字、數據、圖片」，全自動轉化為具備商業變現能力的高價值視覺 Web 應用程式**。

我們不再將其定位為傳統的通用文書軟體，而是「情境驅動的 Web App 引擎」。一個檔案就是一個專注解決特定問題的獨立應用站點（例如「預算規劃器」、「募資簡報工作站」或「商務合約編輯器」），直接賦能行銷、募資、合約、數據決策等商業場景，拒絕為無用的介面切換與過剩的選單買單。

## 核心理念

資料不只是冷冰冰的數字，它應該自帶語意（Markdown）、結構（JSON）與呈現方式（HTML）。系統將運用強大的公式：**網頁最終呈現 = (內容語意 × 模組化設計) + (數據結構 × 排版演算法)**，提供極速的視覺體驗與雙向資料連動。

**一個 `.itw` 檔案 = 一份完整、可讀且自帶設計系統的高級文件**，無需伺服器或第三方依賴，打開瀏覽器即刻產出專業級內容。

## 壹、 核心設計改進 (v2.1 強化)

基於架構重構與高價值商業應用定位，v2.1 將全面改變格式底層邏輯與排版機制：

### 1.1 應用導向的單一 Web App 畫布 (App-Driven Canvas)
*   **破除通用與分頁切換迷思**: 捨棄舊有試圖在單一畫面內塞入「文件、試算表、簡報」三種檢視分頁的臃腫設計。以「Web App」的思維重新定義：當使用者載入 `budget.itw` 時，畫面就是一個專為預算規劃打造的應用系統；載入 `pitch.itw` 時，就是一個專注於募資提案的沉浸式編輯器。不需要切換視圖（View Modes），檔案本身即是解決該問題的專屬應用程式。
*   **跨格式資料融合綁定**: 透過前端狀態管理實現無縫即時重繪。在一份專注於財報的應用畫布中，可以將 CSV 表單計算與 Markdown 產出的動態圖表完全融為一體，實現正在意義上的單一畫布融合，無需任何額外的複製或切換。

### 1.2 增強型純文字容器與本地圖片內嵌

在堅持「單一純文字檔案」以確保高互操作性與版控友好的前提下，系統引入了以下核心機制來解決傳統 Markdown 的痛點：

*   **維持單檔純文字 (`.itw`) 與 Base64 圖片區塊**: 
    *   **傳統痛點**：市面上的 `.docx` 或 `.pptx` 實際上是 ZIP 壓縮包，這類架構會導致檔案喪失純文字可讀性，且無法輕易用 Git 進行逐行版本比對。
    *   **ITW 解法**：徹底放棄 ZIP 容器方案。使用專屬的 `---IMG:id---` 區塊，將介面上拖曳放入的本地圖片自動轉換成 Base64 文字編碼儲存於檔尾。這既能確保圖片跟著檔案走，又能保有 Git 友好的純文字特質。
*   **跨平台圖片內嵌語法 (`itw:img-id`)**: 
    *   **傳統痛點**：Markdown 極度依賴外部絕對路徑或網址，一旦單獨將檔案傳輸到另一台設備，圖片經常會破圖。
    *   **ITW 解法**：在 MD 內文中使用特製語法 `![alt](itw:img-id)` 進行索引。渲染引擎遇此標籤時，會無縫從同一檔案底部的 Base64 區塊抓取資料並顯示，達成真正的「單檔自包含 (Self-contained)」，永不破圖。
*   **Slide 獨立區塊化與佈局解耦 (`---SLIDE:N---`)**: 
    *   **傳統痛點**：Markdown 原生為「由上往下連貫不斷」的文本流，無法滿足簡報場景中「單張投影片需要獨立排版空間」的設計訴求。
    *   **ITW 解法**：引入 `---SLIDE:N layout=X---` 標籤。這如同在純文字中立起分隔牆，將各區塊內容分別封裝為獨立的容器。使每一頁投影片徹底脫離一般文本流的限制，讓排版引擎能依據設定套用滿版、網格等完美的響應式 (RWD) 動態佈局。

### 1.3 高價值動態排版與語意演算法 (Auto-Layout Engine)
*   **內容解析與標籤化**: 系統即時辨識數據層級（KPI vs 輔助文字）、產業情境，自動帶入風格配置，如「募資簡報的高亮關鍵字金流」或「法律文件的嚴謹高對比度」。
*   **多媒體與無障礙自動校調**: 自動進行圖片主旨裁切與底色偵測反白，保證 WCAG 對比度。

## 貳、 底層技術研發 — 拆解傳統工具的黑箱

> 為了達成情境驅動的 Web App 目標，我們拒絕模仿市面軟體的龐大通用介面。本章節探討如何汲取「辦公室套件」、「現代化編輯器」的最強大底層邏輯，並將其重組為能動態生成單一情境 Web App 的功能模塊。

### 2.1 傳統辦公套件技術拆解 (Deconstructing Office Suites)
*   **Word 文本邏輯**: 擷取其段落與樣式分離的概念，但徹底拋棄臃腫的 UI 與封閉的 OOXML 格式，改以人機皆可讀的純 Markdown 作為文本操作核心。
*   **Excel 算力繼承**: 擷取其強大的公式計算能力。核心採用純 CSV 儲存數據，並透過瀏覽器端 HyperFormula 引擎提供算力，確保數據可以直接在特定的應用（如財報網頁或動態儀表板）中被圖表實時調用，而不是丟給用戶一張空白網格。
*   **PowerPoint 敘事結構**: 借鑑其區塊敘事思維與視覺張力（透過 `---SLIDE:N---`），但棄用死板不兼容網頁的像素絕對定位，全面導入 CSS Flexbox/Grid 演算法，實現 100% 完美的響應式 (RWD) 模組佈局。

### 2.2 現代編輯器與區塊引擎的精華萃取
*   **Typora & iA Writer**: 萃取其「所見即所得 (WYSIWYG)」與「高沉浸焦距寫作 (Focus Mode)」。當這份 `.itw` 被判定為「合約文件」的應用時，系統便會主動隱藏無關的圖表按鈕，提供極致專注的純文字環境。
*   **Notion 風格的 Block 架構**: 將底層資料抽象化為 Block（文字段落、表格、資料區塊等皆為 Block）。再由前端渲染引擎依據該 Web App 賦予的情境（預算、簡報、行銷文件），自動決定如何動態排列與設計這些 Block。
*   **底層編碼框架**: 針對需要直接修改源始碼機制的 Power User，內建 CodeMirror 6 作為 Source Mode 的高效引擎。

## 參、 渲染器架構演進與設計決策

本系統在開發初期歷經了架構的迭代，透過驗證不同設計模式，最終確立了目前「情境驅動 (Context-Driven)」的核心走向。

**曾嘗試之架構方案：通用三分頁介面**
在原型階段，編輯器曾嘗試採用「將單一檔案硬性拆分為三種視圖 Tab (MD/CSV/HTML)」的通用佈局，並依賴靜態的模板進行填空。此方案最終遭否決，主因如下：
*   **資料孤島效應**：簡報視圖與文字視圖無法有效引用試算表 (CSV) 的動態數據，跨格式連動極度困難，且圖片無法單檔內建。
*   **介面體驗割裂**：使用者為了解決單一商業問題（如編列行銷預算），卻被迫在三個看似獨立的軟體視圖中反覆切換，增加無謂的認知負擔。

**最終確立架構方案：情境驅動的單一 Web App 渲染器**
為徹底解決資料耦合與多餘的操作，系統底層轉向**「情境應用 Web App 生成器」**的設計模式：
1.  **資料本質宣告**：透過在檔案底層新增 `---IMG:id---`、`---SLIDE:N---` 和 `---THEME---` 等屬性標籤，系統解析後能精準判定該資料容器的「應用情境」（如一份專屬的「預算規劃」專案）。
2.  **動態專屬 UI 生成**：前端引擎讀取解析後，不再提供通用工具列，而是直接將整個編輯畫布建構為**最符合該情境操作的專案應用程式介面**（例如載入完整的 DataGrid 與即時報表系統）。
3.  **單一解題環境**：全面捨棄傳統的「文書/簡報模式」切換開關，讓編輯器成為一個純粹的解題平台，達成完全專注、客製化的互動體驗。

## 肆、 情境驅動的 Web App 引擎實作架構

### 4.1 情境式編輯體驗：專注解決單一問題
使用者登入後面對的，是完全針對其任務（如排版簡報、管控預算）所客製化的單一視覺化 Web App 畫布（App Editor）：
1.  **視覺與源碼雙向同步**: 透過 `serializer.js` 與 `parser.js` 實現 Block DOM 與純文字底層的零延遲轉換。
2.  **沉浸式動態數據**: 在處理具體應用（如財報網頁）時，內建引入 ECharts 配合 Scroll-Trigger，讓枯燥數字轉化為動態趨勢圖，並且當背後數據更動時，前端圖表於 1 秒內無縫重繪。
3.  **封裝為單一應用**: 徹底移除無意義的跨軟體架構介面（Word/Excel/PPT 切換Tab）。打開對應的檔案，Toolbar 與選單即刻收斂為該場景專屬的最佳解題環境，使用者不必思考「我要用什麼軟體開啟這個檔案」，檔案本身就是最好的軟體介面。

### 4.2 核心技術棧
| 功能 | 選用方案 |
|------|---------|
| 安全過濾 | DOMPurify 3.x |
| Markdown 渲染 | marked.js |
| CSV 解析與試算 | PapaParse + HyperFormula |
| 動態數據視覺化 | Apache ECharts / D3.js |
| 原始碼編輯 | CodeMirror 6 |

## 陸、 檔案結構與模組規劃

```
editor/
├── core/
│   ├── parser.js                 ← .itw → AST 解析器
│   └── serializer.js             ← AST → .itw 反向序列化
├── engine/
│   ├── block-engine.js           ← 核心 Block Editor 引擎
│   └── blocks/                   ← 各類型區塊元件 (MD/CSV/JSON/HTML)
├── renderers/
│   ├── budget-app-renderer.js    ← 預算規劃專用 Web App 渲染引擎 (範例)
│   ├── pitch-deck-renderer.js    ← 投資簡報專用 Web App 渲染引擎 (範例)
│   └── contract-renderer.js      ← 合約文件專用 Web App 渲染引擎 (範例)
├── ui/
│   ├── app-shell.js              ← 根據 App 情境自動載入合適 Toolbar 的框架
│   └── source-toggle.js          ← </> 原始碼切換
└── io/
    ├── file-system.js            ← 檔案讀寫
    └── indexeddb.js              ← 草稿暫存
```

## 柒、 開發藍圖 (Roadmap v2.1)

1.  **Phase 1: 格式容器升級與 Web App 環境融合**: 重寫底層 Parser 支援 `---IMG:id---` 圖片 Base64 解析與上傳；廢除三分頁硬框架，重新設計為根據不同情境範本，能動態推演出「專一應用 Web App 畫布」的生成機制。
2.  **Phase 2: 情境獨立區塊與排版引擎**: 實作針對簡報體驗的 `---SLIDE:N---` 和文件類自適應的 Flexbox / Grid 動態排版機制。
3.  **Phase 3: 專案表單與資料雙向綁定**: 針對預算表與追蹤表等應用，導入 HyperFormula 與 ECharts，確保單一應用內資料與視覺體驗的動態連動無縫接軌。
4.  **Phase 4: 商業情境智能加值**: 針對不同應用模組（如行銷傳單、募資簡報），實作專屬語意辨識的高亮、圖片主旨分析與對比度微調，並確保載入與計算效能（Lighthouse > 90, <1.5s）。
5.  **Phase 5: 高階體驗打磨**: 整合 Focus Mode、自製設計系統庫切換，與 PDF/HTML 匯出。
