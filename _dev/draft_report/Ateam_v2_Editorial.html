Product Strategy & Decision Logic

產品經理 - 戰略藍圖與決策鎖定

頂尖 PM 不提供「功能清單」，而是提供「權衡結果 (Trade-offs)」。每一份戰略文件都是為了在資源有限的情境下，保護團隊免於陷入無止盡的開發發散 (Scope Creep)。

Product Strategy (產品戰略文件)

CORE NODE

「戰略是選擇不做什麼。我們定義競爭邊界，確保每一行代碼都符合 3 年後的市場定位。」

核心架構

  • • Playing Field: 明確定義絕對不碰的紅海市場。
  • • Value Guardrail: 即使損失流量，也絕不妥協的使用者價值底線。

方法論應用

OST 機會樹 Wardley Map
⚠️ 品質警戒：平庸戰略的陷阱

避開「充滿形容詞」的文件。如果你的戰略文件中，把目標換成競爭對手也完全正確（例如：提升客戶滿意度），那這不是一份戰略，而是一份願望清單。高品質戰略必須具備「排他性」。

PRD / Spec (需求規格交付)

OPERATIONAL

PRD 是工程團隊唯一的真相來源。它不只解決 Happy Path，更預判了 90% 的例外與邊界情況。

頂尖標準

  • • Edge Case Coverage: 包含離線、超量、逾時等所有 Fallback 邏輯。
  • • Telemetry Spec: 上線第一秒就必須具備的商業數據追蹤規格。

PRO TIP: 逆向工作法

建議採用 Amazon 的 PR/FAQ 格式。在寫第一行 Spec 前，先寫好給使用者的「發佈新聞稿」與「常見問題集」，確保產品價值在啟動前已被驗證。
