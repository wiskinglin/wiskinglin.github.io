## 🎯 修正後的重構策略

### 風險評級

根據連結衝擊分析，搬移檔案的**實際風險遠低於預期**：

| 操作 | 需修改的檔案 | 需修改的連結數 | 風險 |
|------|-------------|--------------|------|
| 搬移根目錄報告到 `reports/` | 只需改 `index.html` | 9 個 href | 🟢 低 |
| 搬移 `pdf.html` + `ebook.html` 到 `tools/` | 只需改 `index.html` | 2 個 href | 🟢 低 |
| 搬移 `top50/` 到 `showcase/top50/` | 需改 `index.html` + `top50/index.html` | 1 + 2 = 3 個 href | 🟢 低 |
| 搬移 `layout/` 到 `_dev/` | **0 個** | **0 個** | 🟢 零風險 |
| 搬移退役檔案到 `_retired/` | **0 個** (因為首頁本來就沒連結) | **0 個** | 🟢 零風險 |

> [!IMPORTANT]
> **核心發現**：因為所有報告/工具頁面都是「葉節點」（不含對外連結），整個專案的連結拓撲是**星狀結構**（star topology），`index.html` 是唯一的中樞。搬移任何葉節點只需要修改中樞（`index.html`）的連結即可，不會產生連鎖反應。

### 方案 A：最小風險方案 ⭐ 推薦

**原則**：只做「不需要修改任何 HTML 內部連結」的操作。

```
wiskinglin.github.io/
├── index.html                          # 不動
├── [所有 Active 報告 & 工具 HTML]       # 不動（留在根目錄）
├── top50/                              # 不動
│
├── _retired/                           # 🆕 退役區（零風險）
│   ├── cv.html                         # ← git mv
│   └── 20260319_zingala.html           # ← git mv
│
├── _dev/                               # 🆕 重新命名開發區（零風險）
│   ├── reports/                        # ← layout/report/
│   ├── tools/                          # ← layout/crafting/
│   └── personal/                       # ← layout/personalWeb/
│
├── docs/                               # 🆕 專案文件（已建立）
├── assets/                             # 🆕 未來共用資源
└── .agents/
```

**需修改的檔案：0 個**  
**風險：🟢 零**

> 這個方案只做「組織性搬移」：把退役頁面明確歸檔，把 `layout/` 更名為語義更清晰的 `_dev/`。根目錄的 Active 內容完全不動，index.html 不需要任何改動。

### 方案 B：中度改造方案

在方案 A 基礎上，加上根目錄報告搬移到 `reports/`。

```
wiskinglin.github.io/
├── index.html                          # ⚠️ 需修改 11 個連結
├── reports/                            # 🆕 報告歸類
│   ├── 2026.html
│   ├── WebUX.html
│   ├── ... (9 個 Active 報告)
│   └── ebook.html (如歸為報告類)
├── tools/                              # 🆕 工具歸類
│   ├── pdf.html
│   └── ebook.html (如歸為工具類)
├── showcase/                           # 🆕
│   └── top50/                          # ⚠️ top50/index.html 需改 2 個連結
├── _retired/
├── _dev/
├── docs/
└── assets/
```

**需修改的檔案：2 個**（`index.html` + `top50/index.html`）  
**需修改的連結：13 個**  
**風險：🟡 低～中**（數量可控，但需仔細驗證）

### 方案 C：完整重構（原始提案）

即 Round 1 提出的完整方案，含子目錄分類、重新命名檔案等。

**需修改的檔案：2 個**（同方案 B）  
**需修改的連結：13 個 + 可能的重命名影響**  
**風險：🟠 中**（加上重命名檔案後，如果未來有外部連結或書籤會失效）
